# Skill: to-prd

Use quando a tarefa for maior que uma PR pequena ou envolver nova capacidade de produto, nova rotina de playtest, biblioteca visual, mudança de pipeline ou arquitetura relevante.

## Procedimento

1. Leia `AGENTS.md`.
2. Leia `docs/LLM_CONTEXT.md`.
3. Leia `docs/ESTADO_ATUAL.md` e `docs/ROADMAP.md`.
4. Não implemente ainda.
5. Escreva um PRD curto.
6. Verifique se a iniciativa respeita a prioridade atual: baseline visual real e playtest antes de features teóricas.
7. Indique se deve seguir para `to-issues`.

## Estrutura do PRD

- Título.
- Objetivo.
- Problema.
- Não-objetivos.
- Cenário de uso.
- Escopo funcional.
- Escopo técnico.
- Critérios de aceitação.
- Riscos e trade-offs.
- Validação.
- Sugestão de decomposição.

## Guardrails

- Não criar especificação gigante.
- Cada critério deve ser verificável.
- Rebaixar prioridade se não ajudar caso, solvabilidade, diversão, facilitador, baseline ou playtest.

## Resposta final

Entregue o PRD e indique a próxima skill recomendada.
